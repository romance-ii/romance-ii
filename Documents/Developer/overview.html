Appius Claudius Caecus

First: Apologies. The documentation for this application is, at best, fragmentary.

Appius Claudius Cæcus is a videogame server, and part of the Romance application stack. Appius provides a game server and a number of facilities that can be shared by Java-based client applications.

Appius was initially developed for the work-in-progress game Siderēs, but most development at this point is being done for the principal commercial sponsor of Appius, Res Interactive, LLC, principally for the game Tootsville™.

Starting Point Vocabulary

A User is a player of the game. For COPPA registrations, a Parent record exists to give authorization. Both a User and a Parent are instances of an AbstractPerson, with a few basic methods.

A Zone contains more than one Room; except for special zones, whose names begin with a '$', which are hidden from logins and serve special purposes. The Zone "$Eden" is used for handling logins and giving players a list of zones. Each Zone can be hosted on a different server. Zones are all instances of the same original game universe.

A World is a collection of Rooms within a Zone.

A Room is a region of space in which events propagate. Events do not generally pass from one room to another (e.g. speech; game actions; &c.).

A StoreItem is a template item; an InventoryItem is an instance of an item that is owned by an user.

An Event is a specific action in-world, which can take a period of time, such as a minigame, combat, purchase, or other activity. Events are started, proceed through their lifetime, and are ended. Events have Outcomes, which reward the creator of the event in some way (e.g. giving a currency or item reward for completing a minigame). Events also have high score lists.

An Action is what most programmers would call an “event,” but that would lead to too much vocabulary confusion. Actions are registered through the Quaestor, and can fire off script actions asynchronously.

DataRecord model

Most objects in the game world are instances of DataRecord. This provides an interface for loading and saving objects to/from some kind of backing-store. Typically, this will be an SQL database, but any kind of storage or communications service is permissible.

A DataRecord is loaded and saved by a RecordLoader bound to its class type through the Java “generics” system.

The RecordLoader for any class can be specified in the configuration file. Default SQL-backed RecordLoaders are provided for most classes.

The “universal factory” for DataRecord-based objects is the Nomenclator class. There are specific methods for instantiating users, because special handling is required for NPC's and such.
